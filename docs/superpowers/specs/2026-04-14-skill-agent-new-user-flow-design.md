# OpenMy Skill Agent 新用户真实录音自动收尾设计

## 背景

现在的新用户链路，前半段已经比较顺：
- 能自动补基础资料
- 能推荐一个真的能用的本地转写路子
- 能先跑演示录音
- 能把真实录音推进到转写、清洗、切段

真正的体验缝在后半段：
- 真实录音跑到后面整理时，系统还会停下来抛选择题
- 这会打断新用户的节奏
- 用户会觉得“怎么前面都自动了，后面又让我选”

这次设计只解决这一件事：
**真实录音进入后半段整理时，Skill Agent 默认继续做完，不再把选择题甩给新用户。**

## 目标

把新用户真实录音链路改成下面这种体验：

1. 用户把录音交进来
2. 系统自动跑前半段
3. 前半段完成后，系统继续自动跑后半段
4. 过程中按阶段给短句进度
5. 如果后半段失败，先明确前面已经做好了什么，再用人话说明后面没做完

## 非目标

这次不做这些事：
- 不改转写模型本身
- 不改网页布局
- 不改真实音频上传协议
- 不改蒸馏和提取的底层算法
- 不做新的设置页开关

## 用户体验规则

### 1. 总原则

新用户一旦已经把真实录音交进来，后面的整理默认继续执行。

只有在用户明确说“别继续”或者明确说“我想自己配 key”时，才停下来。

### 2. 进度播报规则

用分批汇报，不用长段说明，不用技术黑话。

#### 阶段一：开始处理
- 录音收到了，我开始跑了。

#### 阶段二：前半段完成
- 转写、清洗、切段已经好了。

#### 阶段三：后半段继续
- 我正在替你整理后面的结果。

#### 阶段四：全部完成
- 已经整理好了，你现在可以直接看日报。

### 3. 后半段失败时的说法

如果后半段失败，不再弹“双选项”给新用户。

改成固定口径：
- 前面已经做好了，后面整理没做完。
- 你现在先能看前半段结果。
- 我可以继续帮你补后面的整理。

这里要注意：
- 先说已经完成的部分
- 再说没完成的部分
- 最后给一个明确下一步

### 4. 禁止口径

下面这些话不该再直接抛给新用户：
- 方式 A / 方式 B
- LLM key
- provider
- pending / submit
- 花 agent token
- 结构化提取

如果必须提，也只能翻成人话，而且不能当主句。

## 行为设计

### 入口条件

当满足下面条件时，自动进入“后半段继续做完”模式：
- 这是新用户链路
- 真实录音已经开始处理
- 前半段已经完成
- 后半段还没完成
- 用户没有明确要求停下

### 停止条件

只有下面几种情况才停：
- 用户明确说不要继续
- 用户明确说自己要配云端路线
- 后半段失败，且当前轮已经试过一次
- 缺少继续执行所必需的输入，而且系统自己补不上

### 失败处理

后半段失败时：
1. 保留当前已完成的前半段结果
2. 不回滚前半段
3. 给用户一句明确结果
4. 把下一步指向“我继续补后面的整理”

## 结构设计

### Skill Agent 层

Skill Agent 负责三件事：
1. 判断现在是不是新用户真实录音链路
2. 选择回复口径
3. 决定是否自动继续执行后半段

它不需要把内部两条技术路线暴露给用户。

### 执行层

执行层仍然可以保留现有两种后半段做法：
- 直接走项目里已有的自动整理路线
- 或走代理代做路线

但对用户来说，这两条要收口成同一种体验：
**“我继续替你做完。”**

### 状态层

状态里至少要能区分：
- 前半段进行中
- 前半段已完成
- 后半段进行中
- 后半段已完成
- 后半段失败

因为进度播报要依赖这些状态切换。

## 数据流

### 成功链路

1. 录音进入处理
2. 转写完成
3. 清洗完成
4. 切段完成
5. Agent 判定进入自动收尾模式
6. 后半段整理完成
7. 返回日报结果和一句总结

### 失败链路

1. 录音进入处理
2. 转写完成
3. 清洗完成
4. 切段完成
5. 后半段整理失败
6. 保留前半段结果
7. 返回“前面已完成，后面没做完”的结果

## 回复模板

### 成功模板

#### 开始
录音收到了，我开始跑了。

#### 前半段完成
转写、清洗、切段已经好了。

#### 后半段进行中
我正在替你整理后面的结果。

#### 全部完成
已经整理好了，你现在可以直接看日报。

### 失败模板

前面已经做好了，后面整理没做完。
你现在先能看前半段结果。
我接着帮你补后面的整理。

## 测试设计

### 一、对话场景测试

至少补这几类：
1. 新用户选演示录音
2. 新用户切到真实录音
3. 真实录音前半段完成后，Agent 自动继续
4. 后半段失败时，回复不出现技术黑话
5. 后半段失败时，回复先说“前面已经做好了”

### 二、状态测试

至少验证：
1. 前半段完成后能进入后半段自动执行
2. 后半段失败不会把整天状态误写成全失败
3. 前半段结果会保留下来

### 三、口径测试

重点卡这些：
- 不出现“方式 A / 方式 B”
- 不出现“token”
- 不出现“pending / submit”
- 不要求用户二次做选择

## 风险

### 风险一：自动继续后，用户不知道系统还在忙
解决：必须分批汇报，不许静默太久。

### 风险二：失败提示太技术
解决：统一失败模板，先说已完成部分，再说未完成部分。

### 风险三：实现时把“自动继续”做成死循环
解决：失败最多只自动尝试一轮，之后进入明确失败口径。

## 推荐方案

我推荐直接采用这版：
- **默认自动收尾**
- **分批汇报**
- **失败时说人话，不再甩选择题**

原因很简单：
这是最符合“第一次来用”的体验。用户只需要交录音，不需要懂内部路子。

## 验收标准

下面四条都满足，才算这张设计真正成立：

1. 新用户给真实录音后，不会再被问二选一
2. 前半段做完后，系统会自动继续后半段
3. 过程中会有阶段性短句播报
4. 如果后半段失败，回复是人话，而且先说已经完成的部分
