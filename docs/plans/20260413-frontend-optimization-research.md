## 三、现在真正还没补上的前端缺口

下面这些，是我对照 `docs/plans/codex-pr1.5-ceo-review-additions.md` 后确认的“还没到位”项。

### 缺口 A：首页处理完成通知还没接上

我在生产前端里没有找到：
- 浏览器通知权限申请
- 处理完成后的系统通知
- 完成提示音

这意味着：
- 用户把录音拖进去后，如果切到别的窗口，很容易忘记回来
- 处理结束缺少“叫一声用户回来”的动作

### 缺口 B：Demo 入口还没进网页

资料里要求 Drop Zone（拖拽上传区）下面加“试试 3 分钟演示”，但我在生产前端和接口层没有看到网页侧的 `POST /api/demo` 接入点。

这意味着：
- 没有录音的新用户，还是得自己找素材
- 首次体验门槛偏高

### 缺口 C：耗时/费用预估没在首页出现

资料里明确要在上传后、启动处理前告诉用户：
- 预计耗时
- 预计费用

但当前生产页还没有这一层确认信息。

这意味着：
- 用户不知道“要等多久”
- 也不知道“为什么现在还没出结果”

### 缺口 D：渐进结果还没形成用户可见闭环

虽然现在有进度面板，但“每一步产出立即露给用户看”的体验还没有完整打通。

应该做到的是：
- 转写完成 → 立刻给出初步文字
- 清洗完成 → 自动升级为清洗版
- 蒸馏完成 → 顶部摘要出现
- 提取完成 → 事件/待办/决策补全

现在更像“看进度”，还不是“边跑边看结果”。

### 缺口 E：首页“今天最值得记住的一句话”还没进来

资料里把这块定成首页顶层抓手，但生产前端里我还没看到真正渲染这张卡。

这意味着：
- 处理结果虽然完整，但首页还缺一个“先把最有价值的那一句端到脸上”的入口
- 分享裂变也没法做

### 缺口 F：多文件按日期分组选择还没做成

现在拖多个文件进去后，没有看到“按日期分组，让用户选今天这一组先跑”的网页交互。

这意味着：
- 一次拖很多录音时，控制感不够强
- DJI 这类设备录一堆文件时，体验会乱

## 四、我建议的前端优化优先级

### 第一刀：先补“处理完成提醒”

原因很简单：
- 改动小
- 风险低
- 用户马上能感到“这个东西会叫我回来”
- 不用碰太多共享结构

这刀主要是前端改：
1. 首次请求通知权限
2. 任务从运行中切到完成时弹通知
3. 播一声短提示音
4. 同时刷新首页和侧边栏

### 第二刀：再补“耗时/费用预估”

这是最直接的焦虑消解器。

用户拖文件以后，最想知道的就是：
- 还要等多久
- 会不会很贵

这刀做完，等待就从“瞎等”变成“心里有数”。

### 第三刀：再补“渐进结果预览”

这是体验提升最大的一刀，但会碰共享代码最多，所以不建议现在第一时间下手。

因为它要同时改：
- 后端中间产物写出
- 任务状态接口
- 首页进度面板渲染逻辑

多人并行时最容易撞车。

### 第四刀：首页金句卡片

这块很值钱，但它依赖前面的摘要链路更稳。先把“提醒”和“渐进结果”做好，再做它更稳。

## 五、共享文件风险提醒

当前工作树里这些共享文件已经有未提交修改：

- `app/index.html`
- `app/static/app.js`
- `app/static/style.css`

这说明现在直接往生产前端主文件里继续加功能，**很容易和别的并行改动撞在一起**。

所以我建议当前阶段先按下面顺序走：

1. 先用这份研究笔记把缺口排清楚
2. 再由负责人决定先切哪一刀
3. 真要动 `app/static/app.js` 这类共享文件之前，先做写入协调

## 六、我给 leader（负责人）的简短建议

如果现在只准我选一个“最值得先做”的前端优化切口，我选：

**“处理完成提醒 + 首页自动刷新”**

原因：
- 用户最容易立刻感受到
- 工程面最容易单独切出来
- 对现在这套生产页伤害最小
- 做完就能明显减少“我拖进去了，结果呢？”这种悬空感
